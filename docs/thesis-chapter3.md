# 第三章 系統設計與方法

本章將詳細介紹本研究開發之 MeetLove 系統的整體架構設計、各功能模組的運作原理以及所使用的關鍵技術。整個系統涵蓋了從用戶端的視訊擷取、即時情緒分析，到後端的配對邏輯、資料儲存以及結果輸出等完整的技術流程。

## 3.1 系統架構

### 3.1.1 整體系統架構圖

MeetLove 系統採用前後端分離的設計模式，主要由三大部分組成：用戶端（Client）、伺服器端（Server）以及外部服務（External Services）。整體系統的運作流程如圖 3-1 所示。

![圖 3-1 MeetLove 系統整體架構圖]()

系統的資料流向如下：用戶首先透過瀏覽器存取前端應用程式，前端應用程式透過 WebSocket 與伺服器建立連接，並發送配對請求。伺服器接收到配對請求後，會將用戶加入配對隊列，當隊列中有另一位用戶時，系統會進行配對並建立會話。配對成功後，雙方用戶會透過 WebRTC 建立 P2P（Peer-to-Peer）的視訊連線，實現即時的視訊通話。

在視訊通話的過程中，前端應用程式會同時進行情緒分析，包括透過 MediaPipe 進行的表情分析、透過 Web Audio API 進行的語音分析，以及透過 NLP 進行的文字分析。這些分析結果會即時透過 Socket.IO 傳送給伺服器，伺服器會將這些資料與另一位用戶的資料進行整合，計算出最終的情緒判斷結果，並可選擇性地將結果回傳給用戶。

### 3.1.2 客戶端架構

客戶端採用單頁應用程式（Single Page Application, SPA）的設計，主要使用 React 框架進行開發。React 的元件化架構使得程式碼具有良好的可維護性和可擴展性，同時其虛擬 DOM 機制也能夠提供流暢的用戶體驗。

前端應用程式的核心模組包括：

**通訊模組（Communication Module）：** 負責處理與伺服器的連接，包括 Socket.IO 用於即時通訊以及 WebRTC 用於點對點的視訊傳輸。通訊模組封裝了連接管理、事件處理以及錯誤處理等功能，為上層應用提供簡潔的 API。

**視訊處理模組（Video Processing Module）：** 負責擷取用戶的攝影機和麥克風輸入，並將視訊串流傳輸給另一位用戶。該模組使用了瀏覽器原生的 MediaStream API，能夠支援大多數現代瀏覽器。

**情緒分析模組（Emotion Analysis Module）：** 這是本系統的核心模組，負責即時分析用戶的情緒狀態。情緒分析模組進一步分為表情分析、語音分析以及文字分析三個子模組，各自負責處理對應類型的輸入資料。

**用戶介面模組（User Interface Module）：** 負責呈現系統的各個頁面和元件，包括配對頁面、視訊通話頁面、情緒顯示面板等。該模組使用了現代化的 UI 設計語言，確保良好的用戶體驗。

### 3.1.3 伺服器端架構

伺服器端採用 Python 的 FastAPI 框架進行開發。FastAPI 是一個現代化的 Web 框架，支援非同步程式設計，能夠處理高並發的連接請求。同時，FastAPI 還提供了自動的 API 文件生成功能，大幅簡化了開發和調試過程。

伺服器端的主要模組包括：

**API 路由模組（API Routes Module）：** 負責處理 HTTP 請求，包括用戶認證、配對狀態查詢、會話結果取得等 RESTful API。

**Socket.IO 事件處理模組（Socket.IO Event Handler Module）：** 負責處理 WebSocket 連接和事件，包括用戶連線、配對、排隊、訊號交換等各種即時事件。

**配對服務模組（Matching Service Module）：** 負責管理用戶的配對邏輯，包括配對隊列的管理、配對演算法的實現以及會話的創建和管理。

**情緒分析服務模組（Emotion Analysis Service Module）：** 負責接收客戶端傳來的情緒數據，進行整合分析，並計算最終的情緒判斷結果。該模組實現了我們提出的多模態情緒融合演算法。

**資料管理模組（Data Management Module）：** 負責會話資料的儲存和管理，包括 SQLite（用於 MVP）等。該模組提供了簡潔的資料庫操作介面，支援資料的新增、查詢、更新和刪除。

### 3.1.4 資料流設計

系統的資料流可以分為以下幾個主要階段：

**第一階段 - 連接與認證：** 當用戶訪問系統時，前端會建立與伺服器的 Socket.IO 連接。伺服器會為每個連接分配一個唯一的會話 ID（Session ID），並將其與用戶 ID 關聯起來。

**第二階段 - 配對與連線：** 當用戶發起配對請求時，伺服器會將其加入配對隊列。當隊列中有另一位用戶時，系統會創建一個新的會話，並通知雙方。雙方收到配對成功訊息後，會透過 WebRTC 建立 P2P 視訊連線。

**第三階段 - 視訊與情緒分析：** 在視訊通話過程中，前端會持續擷取視訊和音訊資料，並進行即時的情緒分析。分析結果會定時透過 Socket.IO 傳送給伺服器。

**第四階段 - 結果輸出：** 當會話結束時，伺服器會整合雙方的情緒數據，計算最終的喜好判斷結果，並將結果儲存到資料庫中。用戶可以透過 API 查詢歷史會話的結果。

## 3.2 情緒構成分析模型

### 3.2.1 情緒構成要素定義

根據情緒構成理論（Emotion Construction Theory），情緒並非預先存在的離散類別，而是人類透過對身體感覺、行為反應以及環境脈絡的持續建構而形成的。基於這一理論，我們將情緒構成定義為三個主要要素：表情要素、語音要素以及文字要素。

**表情要素（Facial Expression Component）：** 指透過臉部表情傳達的情緒資訊。根據 Ekman 的基本情緒理論，人類的臉部表情能夠傳達六種基本情緒：快樂、悲傷、恐懼、驚訝、厭惡以及憤怒。透過分析臉部特徵點的位置和變化，我們可以推斷出用戶當前的表情狀態，進而推斷其情緒傾向。

**語音要素（Vocal Expression Component）：** 指透過聲音傳達的情緒資訊。語音中蘊含了豐富的情感資訊，包括語調、語速、聲音能量以及停頓模式等。例如，快速的語速和高亢的語調通常與積極的情緒相關，而緩慢的語速和低沉的語調則通常與消極的情緒相關。

**文字要素（Textual Expression Component）：** 指透過對話內容傳達的情緒資訊。對話中的用字遣詞能夠反映出說話者的情緒狀態，例如使用積極正面的詞語通常與正向情緒相關，而使用消極負面的詞語則通常與負向情緒相關。

這三個要素相互關聯、相互補充，共同構成了人類情緒的完整表達。本研究正是基於這一理解，採用多模態的方法來同時處理這三種要素，以獲得更加準確和全面的情緒評估結果。

### 3.2.2 情緒狀態向量設計

為了能夠標準化地表示和分析情緒狀態，我們設計了一個情緒狀態向量（Emotion State Vector）。這個向量是一個多維的數值向量，其中每個維度代表一種情緒屬性。

對於表情分析，我們使用一個包含七個維度的向量來表示情緒狀態：
- E = [e_happy, e_surprised, e_fearful, e_disgusted, e_sad, e_angry, e_neutral]

其中每個維度的值代表對應情緒的強度，範圍為 0 到 1，所有維度的總和為 1。

對於語音分析，我們使用一個包含四個維度的向量來表示：
- V = [v_pitch, v_rate, v_energy, v_sentiment]

其中 v_pitch 表示語調變化程度，v_rate 表示語速，v_energy 表示聲音能量，v_sentiment 表示語音情感傾向。

對於文字分析，我們使用一個包含三個維度的向量來表示：
- T = [t_positive, t_negative, t_neutral]

這些向量不僅可以用於情緒分類，還可以用於後續的個人情緒構成分析，幫助用戶了解自己的情緒模式和特點。

### 3.2.3 動態情緒變化追蹤機制

情緒並非靜態不變的，而是會隨著時間和環境持續變化。為了捕捉這種動態變化，我們設計了一套情緒追蹤機制，能夠在會話過程中持續記錄和分析情緒的變化趨勢。

在技術實現上，我們採用滑動視窗（Sliding Window）的方法來處理情緒數據。我們會維護一個固定大小的時間視窗（例如最近 30 秒），視窗內的情緒數據會被視為當前的情緒狀態。隨著時間的推移，舊的數據會被移除，新的數據會被加入。透過這種方式，系統能夠敏銳地捕捉到情緒的細微變化，同時又不會因為過於敏感的波動而產生雜訊。

此外，我們還計算了情緒變化率（Emotion Change Rate），這是一個描述情緒變化速度的指標。情緒變化率可以幫助我們識別情緒的轉折點，例如從正向情緒快速轉變為負向情緒，這種資訊對於理解用戶的心理狀態具有重要的價值。

## 3.3 表情分析模組

### 3.3.1 臉部偵測流程

表情分析的第一步是臉部偵測（Face Detection），即從影像中找出人臉的位置和範圍。本研究採用 Google 的 MediaPipe Face Mesh 進行臉部偵測，這是一個高效且精確的解決方案，能夠在瀏覽器中即時運作。

臉部偵測的流程如圖 3-2 所示：

![圖 3-2 臉部偵測流程圖]()

1. **影像輸入：** 從攝影機獲取即時的視訊串流，並擷取其中的單一幀（Frame）進行處理。

2. **臉部偵測：** 使用 MediaPipe Face Mesh 的臉部偵測器（Face Detector）對影像進行掃描，識別出人臉的位置。如果影像中包含多張臉部，系統會選擇面積最大的那一張作為分析的目標。

3. **關鍵點偵測：** 使用臉部特徵點偵測器（Face Landmark Detector）識別臉部上的 468 個關鍵點。這些關鍵點分佈在眉毛、眼睛、鼻子、嘴巴以及臉部輪廓等位置，能夠精確地描述臉部的幾何形狀。

4. **特徵映射：** 將識別出的關鍵點座標映射到標準化的座標系統中，以便後續的處理和比較。

### 3.3.2 表情分類模型

在獲得臉部關鍵點之後，我們需要透過這些關鍵點來推斷使用者的表情類別。本研究採用了一種基於幾何特徵的簡化表情分類方法，雖然不如深度學習模型精確，但能夠在瀏覽器中即時運作，且不需要額外的伺服器運算資源。

表情分類的具體方法是基於以下幾個關鍵特徵：

**嘴巴特徵：**
- 嘴巴開合程度：透過嘴巴上下關鍵點的距離來計算
- 嘴巴寬度：透過嘴巴左右關鍵點的距離來計算
- 嘴角上揚程度：透過嘴巴左右角落與嘴巴中心的相對位置來計算

**眼睛特徵：**
- 眼睛睜開程度：透過眼睛上下眼瞼關鍵點的距離來計算
- 眼睛寬度：透過眼睛左右關鍵點的距離來計算

**眉毛特徵：**
- 眉毛上揚程度：透過眉毛與眼睛的相對位置來計算
- 眉毛皺眉程度：透過兩側眉毛的相對位置來計算

基於這些特徵，我們設計了一個簡單的規則基礎分類器，用於將臉部表情分類為七種基本情緒：快樂、悲傷、恐懼、驚訝、厭惡、憤怒以及中性。分類規則是基於對人類面部表情的觀察和心理學研究的結果制定的。

例如，當嘴巴開合程度較大且嘴角上揚時，系統會判定為「快樂」；當眉毛上揚且眼睛睜大時，系統會判定為「驚訝」；當眉毛皺起且嘴巴緊閉時，系統會判定為「憤怒」。

### 3.3.3 情緒指數計算方式

獲得表情分類結果之後，我們需要將其轉換為一個 0 到 1 之間的情緒指數（Emotion Index），以便後續的整合分析。我們設計了一套情緒指數計算公式：

```
EI = Σ(w_i × p_i) / Σ(w_i)
```

其中：
- EI 是最終的情緒指數（0 到 1 之間）
- w_i 是第 i 種情緒的權重
- p_i 是第 i 種情緒的分類機率

情緒的權重是基於情緒的效價（Valence）來設定的：

| 情緒類型 | 權重 | 說明 |
|---------|------|------|
| 快樂 (Happy) | 1.0 | 最積極的情緒 |
| 驚訝 (Surprised) | 0.8 | 中等積極 |
| 中性 (Neutral) | 0.5 | 中間狀態 |
| 恐懼 (Fearful) | 0.3 | 輕度消極 |
| 厭惡 (Disgusted) | 0.2 | 消極 |
| 悲傷 (Sad) | 0.2 | 消極 |
| 憤怒 (Angry) | 0.1 | 最消極 |

這種設計的優點是直觀且易於理解：情緒指數越高，代表情緒越積極；情緒指數越低，代表情緒越消極。同時，這種連續的數值表示也便於後續的統計分析和視覺化呈現。

## 3.4 語音分析模組

### 3.4.1 語音前處理

語音分析的第一步是語音前處理（Speech Preprocessing），目的是將原始的音訊資料轉換為適合後續分析的格式。

語音前處理的主要步驟包括：

**取樣率轉換：** 將音訊的取樣率統一轉換為 16kHz，這是語音處理中最常用的取樣率，能夠在保持語音品質的同時減少運算量。

**分幀（Framing）：** 將連續的音訊訊號切分為固定長度的幀（Frame），通常每幀的長度為 20-40ms。相鄰幀之間通常會有重疊（一般為 50%），以確保分析的連續性。

**加窗（Windowing）：** 對每一幀應用視窗函數（如漢明窗或漢寧窗），以減少頻譜泄漏和邊界效應。

**預加重（Pre-emphasis）：** 對音訊進行高通濾波，增強高頻部分的能量，使語音的頻譜更加平坦。

### 3.4.2 特徵提取方法

特徵提取是語音分析的核心環節，決定了後續分析的準確性和效率。本研究採用以下幾種主要的特徵：

**MFCC（Mel-Frequency Cepstral Coefficients）：** MFCC 是語音識別和語音情緒分析中最廣泛使用的特徵之一。它模擬了人類聽覺系統對頻率的感知特性，能夠有效地捕捉語音的頻譜包絡資訊。我們通常提取 13 維的 MFCC 特徵。

**色度特徵（Chroma Features）：** 色度特徵描述了音樂和語音中的音高結構，能夠捕捉音調的變化。我們使用 12 維的色度特徵。

**頻譜對比度（Spectral Contrast）：** 這個特徵描述了頻譜中不同頻段的能量對比，能夠反映聲音的音色特性。

**頻譜質心（Spectral Centroid）：** 這是頻譜的重心位置，反映了聲音的「亮度」或「清脆度」。

**過零率（Zero Crossing Rate）：** 這是信號通過零點的頻率，與語音的清濁音特性相關。

### 3.4.3 語音情緒分類

在完成特徵提取之後，我們需要根據這些特徵來推斷語音的情緒類別。本研究採用了一種基於規則的方法，結合了上述多種特徵來進行情緒分類。

語音情緒分類的邏輯如下：

**激動（Excited）：** 高語速 + 高能量 + 高音調變化 → 情緒指數高
**平靜（Calm）：** 中等語速 + 中等能量 + 穩定音調 → 情緒指數中等
**沮喪（Depressed）：** 低語速 + 低能量 + 低沉音調 → 情緒指數低
**緊張（Nervous）：** 不規則語速 + 高能量變化 → 情緒指數偏低

此外，我們還可以結合文字分析的結果來提高語音情緒分類的準確性。當文字和語音的情感傾向一致時，我們可以提高分類結果的信心；當兩者不一致時，我們可以進一步分析或者給出多種可能的分類結果。

## 3.5 文字分析模組

### 3.5.1 對話文字擷取

文字分析的第一步是對話文字擷取（Dialogue Text Extraction），即從視訊對話中獲取文字內容。

由於 MVP 原型系統主要是即時的視訊對話，缺乏文字輸入的管道，因此在 MVP 階段，我們的文字分析模組主要用於以下場景：

**語音轉文字（Speech-to-Text）：** 我們可以使用瀏覽器的 Web Speech API（如果支援）來即時地將語音轉換為文字。這種方法的優點是即時性強，缺點是準確率可能受限於環境雜訊和說話者的口音。

**文字聊天（Text Chat）：** 如果系統提供了文字聊天的功能，文字分析模組就可以直接分析使用者輸入的文字內容。

**後期分析（Post-analysis）：** 在會話結束後，可以對會話的錄音進行離線的語音轉文字處理，然後進行文字分析。

### 3.5.2 情感分析模型

文字情感分析（Text Sentiment Analysis）的目標是判斷文字的情感傾向是正面、負面還是中性。本研究採用了 TensorFlow.js 提供的預訓練情感分類模型，能夠在瀏覽器中直接運行，無需將資料上傳到伺服器。

情感分析的輸出包括：
- 情感類別：positive / negative / neutral
- 信心程度：0 到 1 之間的機率值

此外，我們還實現了一個基於關鍵詞的簡單情感分析方法作為備用。這個方法使用預定義的正向和負向關鍵詞列表，當檢測到關鍵詞時，會相應地調整情感分數。這種方法雖然較為簡單，但在某些場景下（如網路連接不佳時）可以作為備用方案。

### 3.5.3 文字情緒分數計算

獲得文字的情感分析結果之後，我們需要將其轉換為一個標準化的情緒分數：

```
TS = (p_positive × 1.0 + p_neutral × 0.5 + p_negative × 0.0)
```

其中：
- TS 是文字情緒分數（0 到 1 之間）
- p_positive 是正向情感的機率
- p_neutral 是中性情感的機率
- p_negative 是負向情感的機率

這種計算方式的直觀解釋是：當文字完全是正向時，TS = 1.0；當文字完全是中性時，TS = 0.5；當文字完全是負向時，TS = 0.0。

此外，我們還會記錄文字中出現的關鍵詞，以便後續進行更細緻的分析。例如，如果一篇對話中多次出現「開心」、「喜歡」等正向詞語，這可能表示用戶對談話內容有較高的興趣和滿意度。

## 3.6 整合決策演算法

### 3.6.1 權重設計

整合決策演算法是將來自表情、語音以及文字三種不同來源的情感資訊融合為最終情緒判斷的關鍵模組。我們設計了一套基於加權融合的方法，其中各來源的權重是基於以下原則確定的：

**表情（Emotion）：權重 0.4**

表情被賦予最高的權重（0.4），原因是：
- 表情是最直觀且最即時的情緒表達方式
- 人類在溝通過程中會下意識地根據對方的表情來判斷情緒
- 表情分析技術相對成熟，準確率較高

**語音（Voice）：權重 0.3**

語音被賦予中等的權重（0.3），原因是：
- 語音攜帶了大量的副語言（Paralinguistic）資訊，如語調、語速、停頓等
- 語音可以反映一些難以從表情看出的情緒，如疲憊、緊張等
- 語音分析技術已經相當成熟

**文字（Text）：權重 0.3**

文字被賦予中等的權重（0.3），原因是：
- 文字提供了最明確的語義資訊
- 文字分析可以捕捉到一些隱含的情緒，如諷刺、暗示等
- 文字分析的結果通常較為可靠

### 3.6.2 融合策略

在確定各來源的權重之後，我們採用以下融合策略來計算最終的情緒分數：

```
Final_Score = (E × 0.4) + (V × 0.3) + (T × 0.3)
```

其中：
- Final_Score 是最終的情緒分數（0 到 1 之間）
- E 是表情分析的情緒分數
- V 是語音分析的情緒分數
- T 是文字分析的情緒分數

這種加權平均的方法是一種典型的晚期融合（Late Fusion）策略，其優點包括：

1. **簡單直觀：** 易於理解和實現
2. **靈活性高：** 可以根據實際效果調整各來源的權重
3. **容錯性強：** 當某一來源的數據缺失或不可靠時，系統仍然可以正常運作
4. **可解釋性：** 可以清楚地看到各來源對最終結果的貢獻

### 3.6.3 High/Normal/Low 判斷閾值

獲得最終情緒分數之後，我們需要根據預定義的閾值將其轉換為 High、Normal、Low 三種狀態：

| 狀態 | 分數範圍 | 閾值 |
|------|----------|------|
| High | 0.7 - 1.0 | >= 0.7 |
| Normal | 0.4 - 0.7 | >= 0.4 且 < 0.7 |
| Low | 0.0 - 0.4 | < 0.4 |

閾值的設計基於以下考量：

**High 閾值（0.7）：** 當情緒分數達到 0.7 以上時，表示情緒相當積極。我們選擇 0.7 作為閾值，是因為這個數值代表著情緒來源中大多數都呈現正向的傾向，是一個相對嚴格的標準。

**Normal 區間（0.4 - 0.7）：** 這是情緒的中間地帶，代表情緒既非特別積極也非特別消極。選擇 0.4 作為下限，是因為這個數值代表著情緒來源中至少有一半以上維持在中性或正向的狀態。

**Low 閾值（0.4）：** 當情緒分數低於 0.4 時，表示情緒偏向消極。選擇 0.4 作為閾值，是為了確保只有當情緒確實偏負面時才會被判定為 Low，避免過度敏感。

這些閾值並非一成不變，未來可以透過實際使用者的回饋和實驗結果進行調整和優化。

### 3.6.4 自信心計算

最後，我們還計算了一個自信心（Confidence）指標，用於表示判斷結果的可信度。自信心的計算基於以下邏輯：

**數據來源數量：** 當三種情緒來源都可用時，自信心會較高；當只有一種或兩種來源時，自信心會相對降低。

**各來源分數的一致性：** 當表情、語音和文字三種分析結果分數接近時，說明情緒判斷較為一致，自信心較高；當各來源結果差異較大時，可能存在衝突或分析錯誤，自信心會降低。

自信心的計算公式如下：

```
Confidence = (Source_Score × 0.3) + (Consistency_Score × 0.7)

其中：
Source_Score = min(available_sources / 3, 1.0)
Consistency_Score = 1 - min(std(E, V, T) × 2, 1)
```

自信心指標可以用於多種場景，例如：
- 當自信心過低時，可以提示用戶重新進行分析
- 可以在結果呈現時顯示判斷的可信度
- 可以用於過濾不可靠的數據
