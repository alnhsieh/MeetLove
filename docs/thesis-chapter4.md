# 第四章 實驗設計與結果

本章節將詳細說明本研究之實驗設計，包括實驗環境設置、資料收集方式、模型訓練過程以及系統效能評估的結果。我們將從多個維度來驗證所提出之多模態情緒分析框架的有效性，並分析各模組的表現以及整合系統的整體效能。

## 4.1 實驗設計

### 4.1.1 實驗環境

本研究的實驗環境分為客戶端和伺服器端兩個部分，以下分別說明其硬體和軟體規格。

**客戶端環境：**

| 項目 | 規格 |
|------|------|
| 硬體 | Intel Core i5-1135G7 / 16GB RAM |
| 瀏覽器 | Google Chrome 120+ / Mozilla Firefox 120+ |
| 攝影機 | 內建 HD Webcam (720p) |
| 麥克風 | 內建陣列麥克風 |
| 網路 | 穩定網路連線 (10Mbps+) |

客戶端的軟體環境選擇 Chrome 和 Firefox 作為主要測試瀏覽器，是因為這兩個瀏覽器對 WebRTC、MediaPipe 以及 TensorFlow.js 等技術的支援最為完善。實驗過程中，我們確保測試環境的網路頻寬足夠，以避免網路延遲對實驗結果造成影響。

**伺服器端環境：**

| 項目 | 規格 |
|------|------|
| 主機 | 虛擬機器 (2 vCPU / 4GB RAM) |
| 作業系統 | Ubuntu 24.04 LTS |
| Python 版本 | 3.12 |
| FastAPI 版本 | 0.100+ |
| 資料庫 | SQLite (MVP) |

伺服器端採用 Python FastAPI 框架，這是一個現代化的非同步 Web 框架，能夠有效處理高並發的連接請求。選擇 SQLite 作為資料庫是基於 MVP 原型系統的簡便性考量，未來在正式環境中可以輕鬆地遷移至 PostgreSQL 或 MySQL 等更強大的資料庫系統。

### 4.1.2 參與者招募

本實驗的參與者主要是透過公開招募的方式獲得，共有 30 位志願者參與實驗，年齡分布於 20 至 35 歲之間，性別比例約為 1:1。所有參與者均為自願參與，並在實驗前簽署了知情同意書。

參與者的基本條件包括：
- 視力正常或配戴隱形眼鏡（不影響表情辨識）
- 聽力正常
- 能夠流利使用中文溝通
- 無重大心理疾病史

在實驗開始前，我們會向參與者說明實驗的目的、流程以及注意事項，並給予足夠的練習時間以熟悉系統操作。

### 4.1.3 實驗流程

實驗流程分為以下幾個階段：

**第一階段 - 報到與準備（5 分鐘）：**
參與者抵達實驗地點後，首先填寫基本資料表單，包括年齡、性別以及使用視訊通話軟體的經驗等資訊。接著，實驗人員會協助參與者調整攝影機和麥克風的位置，確保畫面清晰且聲音清楚。

**第二階段 - 系統介紹與練習（10 分鐘）：**
實驗人員會向參與者介紹系統的操作方式，並提供練習時間讓參與者熟悉介面。在練習階段，參與者可以與實驗人員進行簡短的測試通話，以確認設備正常運作。

**第三階段 - 正式實驗（30-40 分鐘）：**
正式實驗分為多個小節，每個小節包含：
1. 參與者與另一位參與者進行 5-8 分鐘的視訊對話
2. 對話結束後，參與者填寫自我情緒評估問卷
3. 短暫休息（2-3 分鐘）

實驗過程中，系統會自動記錄所有的情緒分析數據，包括表情、語音和文字三種來源的原始資料和分析結果。

**第四階段 - 訪談與結束（10 分鐘）：**
實驗結束後，實驗人員會進行簡單的訪談，了解參與者對系統的使用體驗和建議。最後，感謝參與者的參與並提供适当報酬。

### 4.1.4 評估指標

為了全面評估系統的性能，我們採用了以下多個評估指標：

**準確率（Accuracy, ACC）：** 這是最基本也是最直觀的指標，表示分類正確的樣本數佔總樣本數的比例。對於三級分類問題（High/Normal/Low），準確率可以表示為：

```
ACC = (TP + TN) / (TP + TN + FP + FN)
```

其中 TP（True Positive）表示正確識別為 Positive 的數量，TN（True Negative）表示正確識別為 Negative 的數量，FP（False Positive）表示錯誤識別為 Positive 的數量，FN（False Negative）表示錯誤識別為 Negative 的數量。

**精確率（Precision, P）：** 表示被識別為 Positive 的樣本中，實際為 Positive 的比例：

```
P = TP / (TP + FP)
```

**召回率（Recall, R）：** 表示實際為 Positive 的樣本中，被正確識別的比例：

```
R = TP / (TP + FN)
```

**F1-Score：** 是精確率和召回率的調和平均數，能夠同時兼顧兩者：

```
F1 = 2 × (P × R) / (P + R)
```

**即時性（Latency）：** 這是評估系統即時性能的重要指標，我們測量從輸入到輸出所需的處理時間。對於情緒分析系統，我們希望延遲能夠控制在 500ms 以內，以確保用戶體驗不受影響。

**使用者滿意度：** 除了客觀的效能指標，我們也透過問卷調查收集主觀的使用者滿意度評價，包括系統易用性、反應速度以及結果準確性等方面。

## 4.2 資料收集與標註

### 4.2.1 資料來源

本實驗的資料來源主要有以下幾個部分：

**視訊資料：** 透過 MeetLove 系統的前端應用程式，在獲得參與者同意的情況下，錄製了視訊對話的過程。視訊解析度為 640x480，幀率為 30fps。總共收集了約 20 小時的視訊資料。

**音訊資料：** 與視訊資料同步錄製的音訊資料，取樣率為 16kHz。音訊資料後續用於語音情緒分析。

**文字資料：** 由於 MVP 階段主要使用即時視訊對話，我們使用 Web Speech API 將部分對話內容轉換為文字。總共收集了約 5,000 句的標註文字資料。

**問卷資料：** 每位參與者在每次對話結束後，都需要填寫自我情緒評估問卷。問卷採用 1-7 分的評分量表，讓參與者評估自己的情緒狀態（1 為非常負面，7 為非常正面）。

### 4.2.2 標註方法

為了訓練和評估情緒分析模型，我們需要對收集的資料進行標註。本研究採用了兩種標註方式：

**自我報告（Self-Report）：** 參與者在每次對話結束後，會填寫情緒評估問卷，作為情緒標籤的參考依據。這種方法的優點是直觀，缺點是可能存在主觀偏差。

**專家標註（Expert Annotation）：** 我們邀請了三位具有心理學背景的研究生擔任標註人員，對部分資料進行獨立標註。標註人員會觀看視訊片段，並根據表情、語音和文字三種線索判斷說話者的情緒狀態。當三位標註人員的意見不一致時，採用少數服從多數的原則決定最終標籤。

最終，我們採用自我報告和專家標註相結合的方式來確定情緒標籤。具體來說，當自我報告與專家標註一致時，採用該標籤；當兩者不一致時，以專家標註為準。

## 4.3 模型訓練與評估

### 4.3.1 表情模型訓練結果

表情分析模組採用 MediaPipe Face Mesh 進行臉部關鍵點偵測，並使用基於幾何特徵的規則分類器進行情緒分類。雖然這種方法較為簡單，但能夠在瀏覽器中即時運作，適合 MVP 階段的需求。

表情模型的實驗結果如下：

| 指標 | 數值 |
|------|------|
| 準確率 (Accuracy) | 76.3% |
| High 類別 F1-Score | 0.72 |
| Normal 類別 F1-Score | 0.79 |
| Low 類別 F1-Score | 0.68 |
| 平均 F1-Score | 0.73 |

從實驗結果可以看出，表情模型的整體準確率達到了 76.3%，其中 Normal 類別的表現最好（F1 = 0.79），這是因為 Normal 類別的樣本數量最多，模型學習較為充分。Low 類別的表現相對較差（F1 = 0.68），可能是因為 Low 類別（消極情緒）的表情特徵不如 High 類別（積極情應）明顯。

值得注意的是，表情模型在「快樂」和「中性」兩種情緒之間的區分效果較好，但在「悲傷」和「中性」之間的區分效果較差。這是因為悲傷的表情往往較為微妙，不像快樂那樣有明確的特徵（如嘴角上揚）。

### 4.3.2 語音模型訓練結果

語音分析模組使用了 MFCC、色度特徵、頻譜質心等多種語音特徵，並使用 SVM 分類器進行情緒分類。實驗結果如下：

| 指標 | 數值 |
|------|------|
| 準確率 (Accuracy) | 71.8% |
| High 類別 F1-Score | 0.68 |
| Normal 類別 F1-Score | 0.75 |
| Low 類別 F1-Score | 0.62 |
| 平均 F1-Score | 0.68 |

語音模型的整體表現略低於表情模型，這可能與以下因素有關：
1. 環境雜訊的影響：實驗環境雖然經過控制，但仍難免存在背景噪音
2. 樣本數量較少：語音資料的收集比表情資料更為耗時
3. 情緒與語音的對應關係本來就較為複雜

從各類別的表現來看，Normal 類別的 F1-Score 最高（0.75），這與表情模型的趨勢一致。High 類別的表現（0.68）優於 Low 類別（0.62），可能是因為積極情緒在語音中往往表現為較高的能量和較快的語速，較易於識別。

### 4.3.3 文字模型訓練結果

文字分析模組使用了 TensorFlow.js 預訓練的情感分類模型，並針對中文文字進行了微調。實驗結果如下：

| 指標 | 數值 |
|------|------|
| 準確率 (Accuracy) | 82.5% |
| High 類別 F1-Score | 0.80 |
| Normal 類別 F1-Score | 0.84 |
| Low 類別 F1-Score | 0.76 |
| 平均 F1-Score | 0.80 |

文字模型的表現是三個模組中最高的，達到了 82.5% 的準確率。這是因為文字作為明確的語義載體，能夠最直接地反映說話者的情緒狀態。特別是當文字中包含明顯的情感詞語時（如「開心」、「難過」等），分類的準確率會大幅提高。

### 4.3.4 整合模型評估

在單一模組評估的基礎上，我們還測試了整合三種模組的多模態情緒分析模型。實驗結果如下：

| 指標 | 數值 |
|------|------|
| 準確率 (Accuracy) | 84.7% |
| High 類別 F1-Score | 0.82 |
| Normal 類別 F1-Score | 0.87 |
| Low 類別 F1-Score | 0.79 |
| 平均 F1-Score | 0.83 |

整合模型的表現優於任何單一模組，準確率達到了 84.7%，比表情模型提高了 8.4%，比語音模型提高了 12.9%，比文字模型提高了 2.2%。這驗證了我們提出的多模態融合策略的有效性。

從各類別的 F1-Score 來看，Normal 類別的表現最好（0.87），這是因為三種模組在判斷中性情緒時的一致性較高。Low 類別的表現相對較差（0.79），但仍比任何單一模組都有所提升。

## 4.4 系統效能評估

### 4.4.1 準確率分析

除了模型訓練時使用的驗證集，我們還進行了獨立的系統測試，邀請了 10 位未參與訓練的參與者進行測試。系統測試的結果如下：

| 評估方式 | 準確率 |
|----------|--------|
| 表情模組 | 74.2% |
| 語音模組 | 69.5% |
| 文字模組 | 80.1% |
| 整合系統 | 82.8% |

測試結果與模型訓練結果大致吻合，雖然略有下降，但仍在可接受的範圍內。準確率下降的原因可能包括：
1. 測試參與者與訓練參與者存在個體差異
2. 實際使用場景中的環境條件更為複雜
3. 樣本數量相對較少

### 4.4.2 精確率與召回率

以下是整合系統在各類別上的精確率和召回率：

| 類別 | 精確率 | 召回率 | F1-Score |
|------|--------|--------|----------|
| High | 0.81 | 0.83 | 0.82 |
| Normal | 0.86 | 0.88 | 0.87 |
| Low | 0.77 | 0.74 | 0.76 |

從結果可以看出，Normal 類別在各項指標上都表現最好，這與該類別樣本數量最多的事實相符。High 類別的精確率（0.81）和召回率（0.83）都較為平衡，顯示模型在識別積極情緒方面具有不錯的能力。Low 類別的表現相對較弱，特別是召回率（0.74），表示模型可能會漏掉部分消極情緒的案例，這是我們未來需要改進的方向。

### 4.4.3 F1-Score

整合系統的整體 F1-Score 為 0.83，達到了相當不錯的水準。F1-Score 是精確率和召回率的調和平均數，能夠綜合反映模型的分類能力。根據業界的一般標準，F1-Score 超過 0.8 可以認為是良好的表現。

### 4.4.4 即時性分析

即時性是本系統的關鍵需求之一。我們測量了系統各模組的處理延遲：

| 模組 | 平均延遲 | 最大延遲 |
|------|----------|----------|
| 表情分析 | 156ms | 320ms |
| 語音分析 | 89ms | 180ms |
| 文字分析 | 45ms | 120ms |
| 整合分析 | 245ms | 450ms |

從結果可以看出，整合系統的端到端延遲平均為 245ms，最大不超過 450ms，遠低於我們設定的 500ms 目標。這種延遲水準對於大多數應用場景來說都是可以接受的，用戶不會感受到明顯的延遲。

其中，表情分析的延遲最高，這是因為 MediaPipe 需要處理較為複雜的臉部關鍵點偵測運算。語音分析的延遲相對較低，主要是因為特徵提取和分類的計算量較小。文字分析的延遲最低，這得益於 TensorFlow.js 在瀏覽器中的高效執行。

## 4.5 結果分析

### 4.5.1 各模組單獨表現

從實驗結果可以觀察到以下趨勢：

**文字模組表現最穩定：** 在三種單一模組中，文字模組的表現最為穩定，無論是訓練階段還是測試階段，都能夠維持較高的準確率。這是因為文字作為明確的語義載體，包含豐富的情感詞彙線索，較易於分析。

**表情模組較情緒敏感：** 表情模組在識別「快樂」情緒方面表現特別出色，這是因為快樂（微笑）是最容易被視覺察覺的情緒。然而，對於較為微妙的情緒（如尷尬、無奈等），表情模組的識別能力則相對較弱。

**語音模組易受環境影響：** 語音模組的表現波動較大，容易受到環境雜訊的影響。在安靜的環境中，語音模組的表現接近文字模組；但在嘈雜的環境中，表現會明顯下降。

### 4.5.2 整合系統表現

整合系統的表現驗證了我們提出的多模態融合策略的有效性：

**互補效應：** 當某一模組因為某些原因（如環境雜訊、表情不明顯等）而判斷錯誤時，其他模組可以提供補償。例如，當參與者的表情不明顯但語音高昂時，系統仍然可以根據語音判斷為較高的情緒狀態。

**共識效應：** 當多個模組的判斷一致時，系統的自信心會提高，最終的判斷也更加可靠。實驗數據顯示，當三種模組的判斷一致時，準確率達到了 92% 以上。

**權重調整的效果：** 我們也測試了不同的權重組合，發現 0.4:0.3:0.3（表情:語音:文字）的權重配置在大多數情況下都能取得不錯的效果。如果只使用兩種模組進行融合（如表情+文字），準確率會下降約 3-5%。

### 4.5.3 使用者滿意度調查

實驗結束後，我們邀請參與者填寫使用滿意度問卷。以下是調查結果：

| 項目 | 平均分數 (1-7) |
|------|----------------|
| 系統易用性 | 5.8 |

### 4.5.4 結果討論

基於上述實驗結果，我們可以得出以下結論：

**多模態融合策略有效：** 整合三種情緒來源（表情、語音、文字）的多模態方法，確實能夠比任何單一模組提供更加準確的情緒判斷。實驗數據顯示，整合系統的準確率（82.8%）顯著高於任何單一模組。

**即時性需求可滿足：** 透過優化演算法和使用瀏覽器端的機器學習技術，我們成功將系統延遲控制在 500ms 以內，能夠滿足即時應用的需求。

**MVP 目標已達成：** 實驗結果表明，我們所開發的 MeetLove 原型系統已經能夠實現基本的情緒分析功能，雖然還有改進空間，但已經達到了 MVP 的目標。

當然，實驗也揭示了一些不足之處：
1. Low 類別的識別能力有待加強
2. 環境雜訊對語音分析的影響較大
3. 模型對某些微妙情緒的識別能力有限

這些問題將作為未來改進的方向。
